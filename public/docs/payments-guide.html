Payments

Scope

The scope of this process is more than simply moving to another payment service provider (PSP). We held over 550k payment unique identifiers (tokens) when this project was executed.


Some of these tokens were unnecessary as they were being duplicated within the same client that would have more than one order. There was also record duplication in our database, where a single payment token was shared between multiple clients.


Before deciding to move to another PSP, we had to find the cause of these inconsistencies in our data, which resulted in us supporting deprecated business logic where the order was the core of the entire system and all the tokenized payments were based on the order owning them.


The solution was to migrate the payment tokens to base it on a client level, which adapts to Company_A's current business logic. However, this was challenging since we were dealing with one of the most sensitive parts of the business.


To test card payments in the environment, use the Adyen documentation.

Getting Started

About

Payavel Checkout is a package that integrates seamlessly within the Laravel ecosystem that simplifies the way you integrate & process payments via your preferred payment service provider. Currently, at Company_A we use this service to process our payments.


The Payavel Documentation is allocated at https://payavel.com/guide.




Tokenizing Payment Methods

Send customer payment method information to your PSP and get a token referencing that payment method.


Request

use Payavel\Checkout\Contracts\Billable;

    /**
     * Store the payment method details at the provider.
     * 
     * @param \Payavel\Checkout\Contracts\Billable $billable
     * @param array|mixed $data
     * @return \Payavel\Checkout\PaymentResponse
     */
    public function tokenizePaymentMethod(Billable $billable, $data)
    {
        return tap($this->gateway->tokenizePaymentMethod($billable, $data))->configure(__FUNCTION__, $this->provider, $this->merchant);
    }

To tokenize a payment method you must provide a Private (https://app.clickup.com/2403958/docs/29bkp-4405/29bkp-11025) instance as the first parameter, followed by an array formatted as follows:


  • type: int The PaymentMethod::class model's type_id.

  • cardholder: string The cardholder's full name.

  • number: string The card's 16-digit pan (15 digits for AMEX).

  • expiry_month: string The card's expiration month (2 digits with trailing 0's).

  • expiry_year: string The card's expiration year (4 digits).

  • security_code: string The card's 3-digit security code (4 digits for AMEX).

  • address: array The card's billing address.


Response

/**
 * Maps details from the tokenizePaymentMethod() response to the expected format.
 *
 * @return array|mixed
 */
public function tokenizePaymentMethodResponse();

To standardize the response data returned from the PSP's gateway you must return an array with the following data:


  • wallet_token: string The Wallet::class model's token.

  • token: string The PaymentMethod::class model's token.

  • type: int The PaymentMethod::class models type_id.

  • details: array The PaymentMethod::class model's details.

    • cardholder: string The cardholder's full name.

    • last_digits: string The card's last 4 digits.

    • expiry_month: string The card's expiration month (2 digits with trailing 0's).

    • expiry_year: string The card's expiration year (4 digits).

Collecting Payments

Request a payment authorization, either by providing the raw payment information or using an existing payment method.


In atomic, we use automatic captures, so the authorize function should automatically capture the payment. There is no need to make a capture request.


Request

use rkujawa\\LaravelPaymentGateway\\Contracts\\Billable;

/**
 * Authorize a transaction.
 * 
 * @param array|mixed $data
 * @param \\rkujawa\\LaravelPaymentGateway\\Contracts\\Billable|null $billable
 * @return \\rkujawa\\LaravelPaymentGateway\\PaymentResponse
 */
public function authorize($data, Billable $billable = null);

Using raw payment method information

You may request payment authorization using raw payment method information by providing an array formatted as follows, along with a billable instance as the second parameter:


  • ordernum: string The order reference (ordernum).

  • description: string The order description.

  • amount: int The authorization amount in USD.

  • payment_method: array The raw payment method information as formatted in the Private (https://app.clickup.com/2403958/docs/29bkp-4405/29bkp-10945) request.

    • type: int The PaymentMethod::class models type_id.

    • cardholder: string The cardholders full name.

    • number: string The cards 16-digit pan (15 digits for AMEX).

    • expiry_month: string The cards expiration month (2 digits with trailing 0s).

    • expiry_year: string The cards expiration year (4 digits).

    • security_code: string The cards 3-digit security code (4 digits for AMEX).

    • address: array The cards billing address.


Using an existing payment method

You may also request a payment authorization using an existing payment method by formatting the array as follows:

  • ordernum: string The order reference (ordernum).

  • description: string The order description.

  • amount: int The authorization amount in minor units (cents).

  • payment_method: \rkujawa\LaravelPaymentGateway\Models\PaymentMethod The PaymentMethod::class model instance.

Response

/**
 * Maps details from the authorize() response to the expected format.
 *

Fetching Transaction Details

Request real-time transaction details from the payment service provider.


Please note that it is a good practice to have real-time transaction data stored locally via webhooks, if supported by the PSP, locally stored data can be mapped into the expected response format.

Request

use rkujawa\\LaravelPaymentGateway\\Models\\PaymentTransaction;

/**
 * Retrieve the transaction details from the provider.
 *
 * @param \\rkujawa\\LaravelPaymentGateway\\Models\\PaymentTransaction $transaction
 * @return \\rkujawa\\LaravelPaymentGateway\\PaymentResponse
 */
public function getTransaction(PaymentTransaction $transaction);

In order to fetch transactions details, you must provide a rkujawa\\LaravelPaymentGateway\\Models\\PaymentTransaction::class model instance.

Response

/**
 * Maps details from the getTransaction() response to the expected format.
 * 
 * @return array|mixed
 */
public function getTransactionResponse();

In order to standardize the response data returned from the PSPs gateway you must return an array with the following data:


  • reference: string The unique identifier referencing the PSPs transaction.

  • parent_reference: string (optional) The original transaction reference.

  • amount: int The transaction amount in USD.

  • status_code: int The transactions most recent status code.

  • details: array The transactions details

    • invoice: string The order reference (ordernum).

    • description: string The order description.

  • created_at: string The ISO standard date and time the transaction was submitted.

  • confirmed_at: string (optional) The ISO standard date and time the transaction was finalized.

Chargeback Transactions

Overview

This document serves as a guide for the chargeback process, offering an overview of transaction details, specific reasons for contestation, and supplementary information.


A chargeback is a reversal of a payment transaction, initiated by the cardholder's bank (issuer) after the customer disputes a charge made to their debit or credit card.


When a chargeback is filed, the funds from the original sale are pulled from the sellers account, and are held while the dispute is investigated. If the seller cannot provide enough evidence to refute the claim, the money is permanently lost, and a chargeback fee is charged. Even if the seller wins, the chargeback still counts against their chargeback ratio. See Private (https://app.clickup.com/2403958/docs/29bkp-5225/29bkp-17345)

Workflow for Disputes

These Diagrams aim to guide the big picture of managing the process.


Logic

Classification

How to classify each order? right now, we're mapping 3 principal situations where the order can be:


  • Client Dispute: Any transaction where a client initiates a chargeback, and it is unclear yet if it is a fraud.

  • Fraud: Regarding the fraud documentation, we are already 99% certain that the transaction is fraudulent due to the reasons outlined in the document.

  • Process Error: This situation happens when the client attempts to process a transaction with their card multiple times, but it fails. As a result, the order number gets created multiple times, but only one transaction goes through. Whether this is a fraudulent activity or not depends on whether the cardholder is the same or not.

Fraud Relation

The Fraud Review page is used in the chargeback process of column 5 where the Fraud Analyst reviews the status. If a VIP Decline occurs, it means the user is attempting to place an order that has been previously flagged as a fraud.


Whenever there is a dispute on a chargeback, all the corresponding orders should be marked as fraud on Kount. This will help us keep a record of all the users who have gone through this process, and we can block them if they attempt to do it again.


The VIP Decline list is a crucial part of the Kount system, which contains a significant amount of fraud account information. This list plays an essential role in identifying fraudulent activities and helps the Fraud Team take necessary action. Though there have been discussions about switching from Kount to Adyen, this topic is not relevant to the current document's scope.



Payment Methods

Tokenization

Company_A will automatically tokenize a payment method after a successful transaction. If the company does not have a primary payment method, it will automatically be linked to the company (order) it is being used for.


  • Users can add, edit & remove payment methods at any time within the multi-view settings section in their dashboard (https://Company_A.cool/dashboard/setting/overview/payments).

  • Users can link & unlink payment methods to/from a company within the single-view settings section in their dashboard (https://Company_A.cool/dashboard/payment/info).

  • Users can add new payment methods during the checkout process for any order.

  • After obtaining a successful transaction of an order for a company that does not have a primary payment method, the payment method used for that order will automatically be set as primary.

    • Rules

    Adding a new payment method

    The user may add a new payment method at any time as long as it is not a duplicate.

    Editing an existing payment method

    The user may edit an existing payment method at any time as long as it belongs to them (not a sub-user).

    Removing an existing payment method

    The user may remove an existing payment method if it belongs to the them (not a sub-user).

    If the payment method is linked to a company with an active subscription as the primary payment method, the user must replace the payment method.

    Linking an existing payment method to a company

    The user may link any of his payment methods to his companies at any time (no restriction).

    If the company to which the payment method is being linked does not have a primary payment method, the one being linked will automatically be set as primary.

    Unlinking a payment method linked to a company

    The user may unlink a payment method from a company if it complies with the following rules:


    1. He is the company owner or the payment method belongs to him.

    2. The company does not have an active subscription or the payment method is not primary.

    Conclusion

    We must do everything in our power to ensure that a company with an active subscription has a valid primary payment method at all times and allows the customer to have complete control over their information.

    Paypal Integration

    Introduction

    The Payment Method Information Display feature is crucial to our payment processing system. It allows users to view detailed information about their payment methods, including credit/debit cards and PayPal accounts. This documentation provides an in-depth overview of the feature, its components, and how to use it effectively.


    The Payment Method Information Display feature serves the following purposes:

    • Display detailed payment method information to users.

    • Differentiate between payment types, such as PayPal and credit/debit cards.

    • Handle cases where payment method information may be missing or unavailable.


    Integration Workflow

    Feature Flags

    To support the integration with PayPal, we introduced feature flags. The database can manage these flags to control the availability of PayPal-related features. These feature flags can be used to dynamically control whether specific features or functionalities related to PayPal are active, providing flexibility in managing your application's behavior without needing to modify the codebase.


    Adyen Card Resource

    The Adyen Card Resource formats and processes payment methods, including credit/debit cards. It interacts with Adyen's API to authorize and capture payments.


    The Adyen Payment Type Mapping defines a mapping between payment types (e.g., Visa, Mastercard) and their corresponding constants for easier processing.
